Introduction

Conventions is an integrator framework of Java EE technologies with the objective to easy the development of enterprise web applications. The framework leverages other frameworks such as Hibernate, Primefaces and CODI to delivery solutions to common problems like cruding, exception handling, true pagination, UI components, internationalization, annotated queries, security etc...

This guide is divided into a getting started to setup the framework plus four parts, the first one walks through the framework architecture(Part I, “Architectural Overview”), the second part shows general utility components(Part II, “General utilities”), the third part is dedicated to examples(Part III, “Examples”) and finaly the last part is reserved to Conventions Scaffolding a code generation tool for Conventions based on JBoss Forge(Part IV, “Conventions Scaffolding”).

1.1.2.  conventions-bom

The BOM(Bill of materials) for Conventions framework specifies the versions of each Conventions related dependencies so if you are using Maven 3, you can easily import conventions-bom into your POM by declaring the following in your depdendencyManagement section:


         



        <dependencyManagement> 

            <dependencies>        

                <dependency>

                    <groupId>org.conventionsframework</groupId>

                    <artifactId>conventions-bom</artifactId>

                    <version>0.9.7</version>

                    <type>pom</type>

                    <scope>import</scope>

                 </dependency>

            </dependencies>

        <dependencyManagement>                 



            

now you dont have to worry about dependency version, you just need to declare the groupId and artifactId of each dependency we described in Section 1.1.1, “Declaring maven dependencies”. The Conventions showcase is using the bom, take a look here for an example .

Tip

you can declare the version of the dependencies, if you do so you are overriding the recommended versions declared in conventions-bom.

2.1.  Bean Types

2.1.1.  BaseMBean

As its name says is the base bean for other conventions controllers and it has the following characteristics:

  • manages an entity ( Section 2.2, “ The Bean Entity ”)
  • has an extensible state ( Section 2.3, “ The Bean State ”)
  • access to a service ( Chapter 3, The Service Layer)
  • provides methods to conventions UI components ( Chapter 4, The View Layer)
  • access to the resourceBundle ( Chapter 7, Internationalization)
  • access to the paginator component ( Chapter 8, Paginator)

Here is a simple BaseMBean: 

  

            import org.conventionsframework.bean.BaseMBean;

            @ViewAccessScoped

            @Named("simpleMBean")    

            public class SimpleMBean extends BaseMBean<Person> {

                

                @Inject 

                public void setService(PersonService personService) {

                    super.setBaseService(personService);

                }

                 /**

                * called by saveButton

                */

               @Override

               public void store() {

                   super.store();

               }


               /**

                * called after 'addButton' is clicked  

                * to decide the navigation

                */

               @Override

               public String afterPrepareInsert() {

                   return null;

               }


               /**

                * called after 'editButton' is clicked  

                * to decide the navigation

                */

               @Override

               public String afterPrepareUpdate() {

                   return null;

               }


               /**

                * called when 'filterButton' is clicked

                */

               @Override

               public void find() {

               }

                

                }

            

Important

To perform datatabase operations you need to set the bean service so the code below is required: 


                @Inject 

                public void setService(PersonService personService) {

                    super.setBaseService(personService);

                }

                    

                

Or you can use @Service annotation:

  

            import org.conventionsframework.bean.BaseMBean;

            import org.conventionsframework.qualifier.Service;

            @ViewAccessScoped

            @Named("simpleMBean")    

            @Service(name="name a named service")      

            public class SimpleMBean extends BaseMBean<Person> {

                    

                     }

                

Tip

  

               @Override

               public void store() {

                   super.store();

               }

                

calls the service store method which will perform a hibernate saveOrUpdate on top of the bean entity( Section 2.2, “ The Bean Entity ”). You may override it in your service store method and put some business logic before saving the entity to the datatabase, we will see how to do that on service chapter( Chapter 3, The Service Layer).

For more examples of base conventions beans look at online showcase examples

2.2.  The Bean Entity

A managed bean usually performs operations on top of a JPA entity, this entity is represented by the entity attribute in BaseMBean 

  

            import org.conventionsframework.bean.BaseMBean;

            import javax.inject.Named;

            import org.apache.myfaces.extensions.cdi.core.api.scope.conversation.ViewAccessScoped;

            @ViewAccessScoped

            public class SimpleMBean extends BaseMBean<EntityClass> {

                 public void someMethod(){

                     EntityClass someVar = getEntity();

                     setEntity(new EntityClass());

                }

                }

        

Tip

some Conventions composite components will act on top of the entity atribute, like editButton, removeButton and so on...

the Entity type can be provided by the following ways:

  • @PersistentClass annotation

  

            @RequestScoped

            @PersistentClass(Person.class)

            public class SimpleMBean extends BaseMBean<EntityClass> {

                 public void someMethod(){

                     Person someVar = getEntity();

                     setEntity(new Person());

                }

                }

        

  • override the create method

  

            @RequestScoped

            public class SimpleMBean extends BaseMBean<Person> {

            

                @Override

                public Person create(){

                  return new Person();

                 }

                 public void someMethod(){

                     Person someVar = getEntity();

                     setEntity(new Person());

                }

            

                }

        

Note

if you dont provide a way to resolve the entity the framework will try to resolve it via reflections but avoid that because in some situations your managed bean will be a CDI proxy (eg. when @SecurityMethod annotation is present) and the reflection will not work.

2.3.  The Bean State

A beanState is represented by the State interface which has only a method String getStateName() that describes the current state of the managedBean, the baseMBean has a built in CrudState which represents states for cruding so when you are editing an entity the bean is in updateState, this state information can be useful in various situations such as hiding some parts of the page or executing some business logig only in determinated state for example.

Tip

BaseMBean changes the state automaticly depending on the UI component is used, for example when you click editButton the beanState in set to CrudState.UPDATE

Below is a example of CrudState to hide blocks of page depending on bean state: 




                     <p:outputPanel rendered="#{empty personMBean.beanState or personMBean.findState}">

                        <ui:include src="listUser.xhtml"/>

                    </p:outputPanel>  

                    <p:outputPanel rendered="#{personMBean.insertState or personMBean.updateState}">

                        <ui:include src="addUser.xhtml"/>

                    </p:outputPanel> 

                    <p:outputPanel rendered="#{personMBean.friendState}">

                        <ui:include src="addFriend.xhtml"/>

                    </p:outputPanel> 

                     

                     

        

Below is the use of bean state in the controller: 



                      public class MyBean extends BaseMBean<Entity,Long>{

                    

                        public void anAction(){

                            if(isFindState()){

                                //if bean is in findState do some stuff

                            }

                    

                            setBeanState(CrudState.UPDATE);//change the bean state

                        }

                     }

                     

        

the code below shows how to create your own States 



                        public enum SecurityState implements State{

                        ADMIN("admin"),GUEST("guest");


                        private final String stateName;


                        SecurityState (String stateName){

                            this.stateName = stateName;

                        }


                        @Override

                        public String getStateName() {

                            return this.stateName;

                        }

This section shows how the framework is integrated with CDI, Hibernate and EJB to provide database operations through Conventions Services which are components that link the framework to the backend infrastructure.

3.1.  What's a Service?

In Conventions Framework services are components that have access to the backend infrastructure, below are some characterists of a service:

  • has access to a helper dao( Section 3.2, “ Where is the Dao? ”)
  • access to an entitymanager via EntityManagerProvider component( Section 3.6, “ Entity Manager Provider ” )
  • its the recommended place to put your business logic to help you on the DRY principle and reuse your services among controllers(and maybe tecnologies)
  • access to a Hibernate session
  • built in and configurable true pagination
  • built in cruding operations

3.2.  Where is the Dao?

Conventions framework does not have an explicit Dao layer(although you can create one), the dao has a composition relashionship with the service layer so each service has a dao.

The dao template used by Conventions services can be found here

3.3. The built in Services

Conventions counts with three extensible built in services templates that diff from each other only by its entityManagerProvider ( Section 3.6, “ Entity Manager Provider ”).

3.3.1.  Stateless Service

Has a StatelessEntityManager provider ( StatelessEntityManagerProvider: ) so your hibernate session/entityManager will be recreated on every request.

3.3.1.1.  Injecting the Stateless Service

To use a service you'll need to inject it(usually in a controller) so we provided three ways to inject a StatelessService

By type: 


                               public class MyBean{   

                                @Inject @PersistentClass(value=Person.class)

                                private StatelessHibernateService myService;  

                               }   

                    

                            

Note

Every service needs an entity to perform database operations, in the above example we specify the entity at the injection point via @PersistentClass annotation where Person class is a JPA entity.
Via qualifier: 


                               public class MyBean{   

                                @Inject @Service(type= Type.STATELESS,entity=Person.class)

                                private BaseService myService;  

                               }   

                    

                            

Note

when using the Service qualifier we are injecting the BaseService interface so we are being type safe.

Also note that in the above example we specify the entity in the service annotation itself so myService is an instance of StatelessService which performs database operations under the Person JPA entity.

By name:

Injection by name is only available with @Service annotation via class level annotation in a Conventions bean Section 2.1, “ Bean Types ” 



                               @Service(name=Service.STATELESS, entity=Person.class)   

                               public class MyBean extends BaseMBean{   

                                

                                  public void aMethod(){

                                    getBaseService();//will return an instance of the StatelessService

                                  }  

                               }   

                    

                            

Note

the name of the service must match the value of the @Named(value="name of the service") annotation of a service.

The names of the built in services are available in the Service class as constants, eg: Service.STATELESS is the name of the StatelessService.

3.3.2.  Stateful Service

Has a StatefulEntityManager provider ( StatefulEntityManagerProvider: ) so your hibernate session/entityManager will be reused as long as the bean this service is injected is alive.

So for example if the StatefulService is injected in a controller with ViewScope your PersistenceContext will be keeped alive(your hibernate session will be open) as long the user stays on the same view no matter how many access to the datatabase he does.

Tip

Stateless and Stateful services can be turned in an EJB when you extend them ( Section 3.4, “ Extending a built in Service ”) by doing that you're turning the methods below to run within an EJB transaction: 

 

                            void store(T Entity);


                            void afterStore(T entity);


                            void beforeStore(T entity);


                            void remove(T entity);


                            void beforeRemove(T entity);


                            void afterRemove(T entity);

                         

                

all other methods are marked as TransactionAttributeType.SUPPORTS

more details at Section 3.4.2, “ Turning your service in an EJB ”

3.4.  Extending a built in Service

The built in services are helper templates that has easy access to the datatabase with handy helper methods but as we mentionated before another advantage of a service is to hold your business logic but you cant do that in a built in service cause we dont know which methods you will need so one way to do that is to extend a built in service or even creating your own service ( Section 3.5, “ Creating your own Service ”).

The first step to extend a service is to create an interface which extends BaseService 

 

                        public interface PersonService extends BaseService<Person, Long>{

              

                        }

                         

        

Note

Where Person is the JPA entity that will be managed by the service and Long is the type of the Person primary key.

Now the service implementation will need to extend a built in service and implement the created interface 

 

                      @Named(value = "personService")

                      @PersistentClass(Person.class)

                      public class PersonServiceImpl extends StatefulHibernateService<Person, Long> implements PersonService {

                

                        //put your business logic here

                        }

                         

        

Note

The @PersistentClass(Person.class) sets the JPA entity that will be managed by the service. Another way to do set the service entity is to implement the getPersistenceClass method in the service 
 

             @Named(value = "personService")

             public class PersonServiceImpl extends StatefulHibernateService<Person, Long> implements PersonService {

                

                   public Class<Person> getPersistenceClass() {

                      return Person.class;

                    } 

                    //put your business logic here

                   }

                

            
Also if you dont specify the PersistenceClass in neither of the above ways Conventions will try to infer it via reflection.

3.4.1.  Injecting the extended service

You can inject your extended Service in the following ways

By type: 


                               public class MyBean{   

                                @Inject 

                                private PersonService myService;  

                               }   

                    

                            

Note

now you dont need to specify an entity cause it was specified in your extended service.
By name:

Injection by name is only available with @Service annotation via class level annotation in a Conventions bean Section 2.1, “ Bean Types ” 



                               @Service(name="personService")//no need to specify the entity here   

                               public class MyBean extends BaseMBean{   

                                

                                  public void aMethod(){

                                    getBaseService();//will return an instance of the PersonServiceImpl

                                  }  

                               }   

                    

                            

Note

the name of the service must match the value of the @Named(value="name of the service") annotation of a service.

3.6.  Entity Manager Provider

The EntityManagerProvider is an important component in the framework architecture, it is the 'guy' who provide instances of the so called EntityManager.

Conventions comes with three built in entityManagerProviders that are used by the built in Services but creating your own provider is very easy and will be demostrated in the next sections.

3.6.1.  The built in providers

StatelessEntityManagerProvider:

is a Stateless EJB that provides the entity manager via a Transaction PersistenContext, it is used by the StatelessHibernateService.

StatefulEntityManagerProvider:

is a Stateful EJB that provides the entity manager via an Extended PersistenContext, it is used by the StatefulHibernateService.

Tip

You rarely will touch in a built in entityManagerProvider code but its important to know what kind of PersistenceContext your service is using and how your entities will be managed so you can solve problems like 'lazyInitialization' accordingly.
CustomEntityManagerProvider:

is a is a dummy provider that you MUST extend to get it working properly, see Section 3.6.2, “ Extending an EntityManagerProvider ”. It is used by CustomHibernateService.

Tip

CustomEntityManagerProvider and CustomHibernateService were designed to work without EJB and are used in Tomcat sample applications.

3.6.2.  Extending an EntityManagerProvider

The built in providers will work as long you have only one persistence unit if you have multiple datatasources in your application you will need to extend an existing provider to specify which datasource the provider will use, to do that you will need:

  • extend an existing provider
  • use the CDI @Specializes qualifier
  • use @ConventionsEntityManager qualifier with the right type

    so if you are extending the StatelessEntityManagerProvider then type=Type.STATELESS

  • implement the EntityManagerProvider interface
  • override the getEntityManager() method.

here is an example: 

  package org.conventionsframework.showcase.provider;


import org.conventionsframework.showcase.controller.ComboMBean;

import javax.enterprise.context.RequestScoped;

import javax.enterprise.inject.Specializes;

import javax.persistence.EntityManager;

import javax.persistence.PersistenceContext;

import org.conventionsframework.entitymanager.CustomEntityManagerProvider;

import org.conventionsframework.entitymanager.EntityManagerProvider;

import org.conventionsframework.qualifier.ConventionsEntityManager;

import org.conventionsframework.qualifier.Type;


/**

 * entityManager provider for CustomHibernateService

 * @see ComboMBean#initList(org.conventionsframework.service.BaseService) 

 * @author rmpestano

 */

@Specializes

@RequestScoped

@ConventionsEntityManager(type=Type.CUSTOM)

public class ShowcaseEntityManagerProvider extends CustomEntityManagerProvider implements EntityManagerProvider {


    @PersistenceContext(unitName="anotherUnit")

    private EntityManager entityManager;//can be provided by an EntityManagerFactory


    @Override

    public EntityManager getEntityManager() {

        return entityManager;

    }


     

}

        

Note

the @ConventionsEntityManager qualifier is needed if you are using a built in service so for exemple when you say your provider is type=Type.CUSTOM you are specifying the entityManagerProvider for CustomHibernateService, but you can extend a built in provider and use it in your own service if do that you wont need the @ConventionsEntityManager but an easy way to do that is creating your own EntityManagerProvider.

Important

When you have multiples services that extend the built in Services keep in mind that your services are sharing the same PersistenceContext(when type=EXTENDED) so if you want a PersistenceContext per Service you must create an EntityManager provider for each service.

3.6.3.  Creating your own EntityManagerProvider

To create your provider you'll need:

  • implement the EntityManagerProvider interface
  • implement the getEntityManager() method.
  • provide the @Named annotation to differentiate your provider from existing ones.
  • inject it in a service you create like in the example below

here is an example: 

 package org.conventionsframework.showcase.provider;


import javax.ejb.Stateless;

import javax.enterprise.context.Dependent;

import javax.inject.Named;

import javax.persistence.EntityManager;

import javax.persistence.PersistenceContext;

import org.conventionsframework.entitymanager.EntityManagerProvider;


/**

 *

 * @author Rafaem M. Pestano - Sep 12, 2012 1:05:46 PM

 */

@Stateless

@Dependent

@Named("myProvider")

public class MyEntityManagerProvider implements EntityManagerProvider{


    @PersistenceContext

    private EntityManager entityManager;

    

    

    @Override

    public EntityManager getEntityManager() {

        return entityManager;

    }


    @Override

    public void setEntityManager(EntityManager entityManager) {

        this.entityManager = entityManager;

    }


}

        

Tip

you also can change the EntityManagerProvider of a built in Service, here is how to:



@Named("personService")

public class PersonServiceImpl extends StatefulHibernateService<Person, Long> implements PersonService {

                 //note that StatefulHibernateService uses StatefulEntityManagerProvider

                 @Inject @Named("myProvider")

                 private EntityManagerProvider entityManagerProvider;//example of created EntityManagerProvider instead of using built in

                 

                 @PostContruct

                 void changeProvider(){

                   super.setEntityManagerProvider(entityManagerProvider);

                 }

                 

                 //OR override getter

               /* @Override

                public EntityManagerProvider getEntityManagerProvider() {

                    return this.entityManagerProvider;

                 } */

               }

                 

            

3.6.4. setting entity manager "by hand"

If you dont want to create or extend an entityManagerProvider but still want one PersistenceContext per service you can set the entityManager by hand in each service, see the code below: 



@Named("personService")

@Stateful//needed if you want to have a persistenceContext           

public class PersonServiceImpl extends StatefulHibernateService<Person, Long> implements PersonService {

                 //note that StatefulHibernateService uses StatefulEntityManagerProvider

                 @PersistenceContext(unitName="myUnit")

                 private EntityManager entityManager;

           

                 @PostContruct

                 void changeEntityManager(){

                   super.getEntityManagerProvider().setEntityManager(entityManager);

                 }

               }
