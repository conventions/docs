Introduction

Conventions is an integrator framework of Java EE technologies with the objective to easy the development of enterprise rich web applications. The framework leverages other frameworks such as Hibernate, Primefaces and CODI to delivery solutions to common problems like cruding, exception handling, true pagination, helper UI components, internationalization, annotated queries etc...

This guide is divided into a getting started to setup the framework plus three parts, the first one walks through the framework architecture, the second part shows general utility components such as annotated queries, i18n, exception handling etc... finaly the last part is reserved to Conventions Scaffolding a code generation tool for Conventions based on JBoss Forge.

Chapter 1. Getting Started

Getting started with Weld Extensions is easy. If you are using Maven, then you can declare a dependency on Weld Extensions (org.jboss.weld:weld-extensions:${weld.extensions.version}, make sure you have the JBoss Maven repository enabled). Otherwise, add the jar to your compile time and runtime classpath.

Most of Weld Extensions has very few dependencies:

  • javax.enterprise:cdi-api
  • org.slf4j:slf4j-api
  • org.jboss.logging:jboss-logging-api
  • javax.el:el-api
  • javax.inject:javax.inject
  • javax.transaction:jta

Tip

The POM for Weld Extensions specifies the versions required. If you are using Maven 3, you can easily import the dependencyManagement into your POM by declaring the following in your depdendencyManagement section: 


         <dependency>

            <groupId>org.jboss.weld</groupId>

            <artifactId>weld-extensions</artifactId>

            <version>${weld.extensions.version}</version>

            <type>pom</type>

            <scope>import</scope>

         </dependency>

Some features of Weld Extensions require additional dependencies (which are declared optional, so will not be added as transitive dependencies):

org.javassist:javassist
Service Handlers, Unwrapping Producer Methods
javax.servlet:servlet-api
Accessing resources from the Servlet Context

2.1.  Bean Types

2.1.1.  BaseMBean

As its name says is the base bean for other conventions controllers and it has the following characteristics:

  • access to a service (link to service layer)
  • manages an entity(link to model layer)
  • provides methods to conventions UI components(link to view layer) such as addButton, editButton, removeButtom, filterButton, clearButton etc...
  • access to the paginator component(helper for true pagination)
  • access to the resourceBundle
  • BeanStates for cruding (link to beanState section)

Here is a simple BaseMBean: 

  

            import org.conventionsframework.bean.BaseMBean;

            @ViewAccessScoped

            @Named("simpleMBean")    

            public class SimpleMBean extends BaseMBean<Person> {

                

                @Inject 

                public void setService(PersonService personService) {

                    super.setBaseService(personService);

                }

                 /**

                * called by saveButton

                */

               @Override

               public void store() {

                   super.store();

               }


               /**

                * called after 'addButton' is clicked  

                * to decide the navigation

                */

               @Override

               public String afterPrepareInsert() {

                   return null;

               }


               /**

                * called after 'editButton' is clicked  

                * to decide the navigation

                */

               @Override

               public String afterPrepareUpdate() {

                   return null;

               }


               /**

                * called when 'filterButton' is clicked

                */

               @Override

               public void find() {

               }

                

                }

            

Important

To perform datatabase operations you need to set the bean service so the code below is required: 


                @Inject 

                public void setService(PersonService personService) {

                    super.setBaseService(personService);

                }

                    

Or you can use @Service annotation:

  

            import org.conventionsframework.bean.BaseMBean;

            import org.conventionsframework.qualifier.Service;

            @ViewAccessScoped

            @Named("simpleMBean")    

            @Service(name="name a named service")      

            public class SimpleMBean extends BaseMBean<Person> {

                    

                     }

Tip

  

               @Override

               public void store() {

                   super.store();

               }

calls the service store method which will perform a hibernate saveOrUpdate on top of bean entity you can override it in your service store method and put some businessLogic before saving the entity to the datatabase.

For more examples of base conventions beans look at online showcase examples

2.2.  The Bean Entity

A managed bean usually performs operations on top of a JPA entity, this entity is represented by the entity attribute in BaseMBean 

  

            import org.conventionsframework.bean.BaseMBean;

            import javax.inject.Named;

            import org.apache.myfaces.extensions.cdi.core.api.scope.conversation.ViewAccessScoped;

            @ViewAccessScoped

            public class SimpleMBean extends BaseMBean<EntityClass> {

                 public void someMethod(){

                     EntityClass someVar = getEntity();

                     setEntity(new EntityClass());

                }

                }

            

Tip

some Conventions composite components will act on top of the entity atribute, like editButton, removeButton and so on...

the Entity type can be provided by the following ways:

  • @PersistentClass annotation

  

            @RequestScoped

            @PersistentClass(Person.class)

            public class SimpleMBean extends BaseMBean<EntityClass> {

                 public void someMethod(){

                     Person someVar = getEntity();

                     setEntity(new Person());

                }

                }

            

  • override the create method

  

            @RequestScoped

            public class SimpleMBean extends BaseMBean<Person> {

            

                @Override

                public Person create(){

                  return new Person();

                 }

                 public void someMethod(){

                     Person someVar = getEntity();

                     setEntity(new Person());

                }

            

                }

            

Note

if you dont provide a way to resolve the entity the framework will try to resolve it via reflections but avoid that because in some situations your managed bean will be a CDI proxy (eg. when @SecurityMethod annotation is present) and the reflection will not work.

2.3.  The Bean State

A beanState is represented by the State interface which has only a method String getStateName() that describes the current state of the managedBean, the baseMBean has a built in CrudState which represents states for cruding so when you are editing an entity the bean is in updateState, this state information can be useful in various situations such as hiding some parts of the page or executing some business logig only in determinated state and so on.

Tip

BaseMBean changes the state automaticly depending on the UI component is used, for example when you click editButton the beanState in set to CrudState.UPDATE

Below is a example of CrudState to hide blocks of page depending on bean state: 




                     <p:outputPanel rendered="#{empty personMBean.beanState or personMBean.findState}">

                        <ui:include src="listUser.xhtml"/>

                    </p:outputPanel>  

                    <p:outputPanel rendered="#{personMBean.insertState or personMBean.updateState}">

                        <ui:include src="addUser.xhtml"/>

                    </p:outputPanel> 

                    <p:outputPanel rendered="#{personMBean.friendState}">

                        <ui:include src="addFriend.xhtml"/>

                    </p:outputPanel> 

                     

                     

        

Below is the use of bean state in the controller: 



                      public class MyBean extends BaseMBean<Entity,Long>{

                    

                        public void anAction(){

                            if(isFindState()){

                                //if bean is in findState do some stuff

                            }

                    

                            setBeanState(CrudState.UPDATE);//change the bean state

                        }

                     }

                     

        

the code below shows how to create your own States 



                        public enum SecurityState implements State{

                        ADMIN("admin"),GUEST("guest");


                        private final String stateName;


                        SecurityState (String stateName){

                            this.stateName = stateName;

                        }


                        @Override

                        public String getStateName() {

                            return this.stateName;

                        }

3.1.  What's a Service?

In Conventions Framework services are components that have access to the backend infrastructure, below are some characterists of a service:

  • has access to an entitymanager via EntityManagerProvider component
  • has access to a Hibernate session
  • built in and configurable pagination method
  • built in cruding operations

3.2.  Where is the Dao?

Conventions framework does not have an explicit Dao layer(although you can create one), the dao has a composition relashionship with the service layer so each service has a dao.

3.5.  Entity Manager Provider

The EntityManagerProvider is an important component in the framework architecture, it is the 'guy' who provide instances of the so called EntityManager.

Conventions comes with three built in entityManagerProviders that are used by the built in Services but creating your own provider is very easy and will be demostrated in the next sections.

3.5.1.  The built in providers

StatelessEntityManagerProvider: is a Stateless EJB that provides the entity manager via a Transaction PersistenContext, it is used by the StatelessHibernateService.

StatefulEntityManagerProvider: is a Stateful EJB that provides the entity manager via an Extended PersistenContext, it is used by the StatefulHibernateService.

Tip

You rarely will touch in a built in entityManagerProvider code but its important to know what kind of PersistenceContext your service is using and how your entities will be managed so you can solve problems like 'lazyInitialization' accordingly.

CustomEntityManagerProvider: is a dummy provider that you MUST extend to get it working properly, see Section 3.5.2, “ Extending an EntityManagerProvider ”. It is used by CustomHibernateService.

Tip

CustomEntityManagerProvider and CustomHibernateService were designed to work without EJB and are used in Tomcat sample applications.

3.5.2.  Extending an EntityManagerProvider

The built in providers will work as long you have only one persistence unit if you have multiple datatasources in your application you will need to extend an existing provider to specify which datasource the provider will use, to do that you will need:

  • extend an existing provider
  • use the CDI @Specializes qualifier
  • use@ConventionsEntityManager qualifier with the right type

    so if you are extending the StatelessEntityManagerProvider then type=Type.STATELESS

  • implement the EntityManagerProvider interface
  • override the getEntityManager() method.

here is an example: 

  package org.conventionsframework.showcase.provider;


import org.conventionsframework.showcase.controller.ComboMBean;

import javax.enterprise.context.RequestScoped;

import javax.enterprise.inject.Specializes;

import javax.persistence.EntityManager;

import javax.persistence.PersistenceContext;

import org.conventionsframework.entitymanager.CustomEntityManagerProvider;

import org.conventionsframework.entitymanager.EntityManagerProvider;

import org.conventionsframework.qualifier.ConventionsEntityManager;

import org.conventionsframework.qualifier.Type;


/**

 * entityManager provider for CustomHibernateService

 * @see ComboMBean#initList(org.conventionsframework.service.BaseService) 

 * @author rmpestano

 */

@Specializes

@RequestScoped

@ConventionsEntityManager(type=Type.CUSTOM)

public class ShowcaseEntityManagerProvider extends CustomEntityManagerProvider implements EntityManagerProvider {


    @PersistenceContext(unitName="anotherUnit")

    private EntityManager entityManager;//can be provided by an EntityManagerFactory


    @Override

    public EntityManager getEntityManager() {

        return entityManager;

    }


     

}

Note

the @ConventionsEntityManager qualifier is needed if you are using a built in service so for exemple when you say your provider is type=Type.CUSTOM you are specifying the entityManagerProvider for CustomHibernateService, but you can extend a built in provider and use it in your own service if do that you wont need the @ConventionsEntityManager but an easy way to do that is creating your own EntityManagerProvider.

Important

When you have multiples services that extend the built in Services keep in mind that your services are sharing the same PersistenceContext(when type=EXTENDED) so if you want a PersistenceContext per Service you must create an EntityManager provider for each service.

3.5.3.  Creating your own EntityManagerProvider

To create your provider you'll need:

  • implement the EntityManagerProvider interface
  • implement the getEntityManager() method.
  • provide the @Named annotation to differentiate your provider from existing ones.
  • inject it in a service you create like in the example below

here is an example: 

 package org.conventionsframework.showcase.provider;


import javax.ejb.Stateless;

import javax.enterprise.context.Dependent;

import javax.inject.Named;

import javax.persistence.EntityManager;

import javax.persistence.PersistenceContext;

import org.conventionsframework.entitymanager.EntityManagerProvider;


/**

 *

 * @author Rafaem M. Pestano - Sep 12, 2012 1:05:46 PM

 */

@Stateless

@Dependent

@Named("myProvider")

public class MyEntityManagerProvider implements EntityManagerProvider{


    @PersistenceContext

    private EntityManager entityManager;

    

    

    @Override

    public EntityManager getEntityManager() {

        return entityManager;

    }


    @Override

    public void setEntityManager(EntityManager entityManager) {

        this.entityManager = entityManager;

    }


}

Tip

you also can change the EntityManagerProvider of a built in Service, here is how to:



@Named("personService")

public class PersonServiceImpl extends StatefulHibernateService<Person, Long> implements PersonService {

                 //note that StatefulHibernateService uses StatefulEntityManagerProvider

                 @Inject @Named("myProvider")

                 private EntityManagerProvider entityManagerProvider;//example of created EntityManagerProvider instead of using built in

                 

                 @PostContruct

                 void changeProvider(){

                   super.setEntityManagerProvider(entityManagerProvider);

                 }

                 

                 //OR override getter

               /* @Override

                public EntityManagerProvider getEntityManagerProvider() {

                    return this.entityManagerProvider;

                 } */

               }

                 

3.5.4. setting entity manager "by hand"

If you dont want to create or extend an entityManagerProvider but still want one PersistenceContext per service you can set the entityManager by hand in each service, see the code below: 



@Named("personService")

@Stateful//needed if you want to have a persistenceContext           

public class PersonServiceImpl extends StatefulHibernateService<Person, Long> implements PersonService {

                 //note that StatefulHibernateService uses StatefulEntityManagerProvider

                 @PersistenceContext(unitName="myUnit")

                 private EntityManager entityManager;

           

                 @PostContruct

                 void changeEntityManager(){

                   super.getEntityManagerProvider().setEntityManager(entityManager);

                 }

               }
